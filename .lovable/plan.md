

# План: Убрать преждевременный перенос — заполнять страницу до орнамента

## Корень проблемы
Пагинация измеряет доступную высоту так:
```
availableHeight = container.clientHeight - 24
```
плюс контейнер имеет `padding: "8px 4px 24px 4px"` (нижний padding = 24px).

Итого «съедается» снизу: `padding-bottom 24px` + `буфер 24px` = **48px**, плюс ещё номер страницы (`bottom: 4px`, высота ~20px) уже учтён в padding-bottom. Получается двойной запас → последняя запись не влезает и переносится, хотя визуально снизу остаётся пустота.

К тому же измеритель использует упрощённый шаблон (без gap, без классов Tailwind у реальных entry), который может занижать высоту слегка — но главная проблема в избыточном запасе.

## Изменение в `src/components/FinalBook.tsx`

### 1. Уменьшить `padding-bottom` контейнера левой страницы (строка 178)
```
padding: "8px 4px 24px 4px"  →  "8px 4px 20px 4px"
```
20px ровно покрывает номер страницы (`— 1 —`, высота ~18–20px), без лишнего запаса.

### 2. Убрать дополнительный буфер из расчёта высоты (строка 45)
```
const availableHeight = container.clientHeight - 24;
       →
const availableHeight = container.clientHeight - 4;
```
`clientHeight` уже **исключает** padding-bottom (20px) — значит номер страницы не пересечётся. Оставляем минимальный 4px буфер на округления браузера.

### 3. Синхронно обновить правую страницу (строка 193)
Чтобы пагинация работала консистентно (измеряется по левому контейнеру, но рендер тоже на правой):
```
padding: "8px 4px 24px 4px"  →  "8px 4px 20px 4px"
```

## Что НЕ затрагивается
- `left/top/width/height` контейнеров — позиционирование уже выровнено по орнаменту
- Данные, `entries`, localStorage
- Логика рендера, реакции, шрифты
- MagicBook, навигация, звуки

## Результат
- Страница заполняется почти до нижнего орнамента
- Запись, которая раньше «не влезала» из-за фиктивных 48px запаса, теперь помещается
- Номер страницы не перекрывается (20px padding-bottom его удерживает)
- Текст не заходит на орнамент (height 58% = до 79% экрана, а орнамент начинается ~80%)
